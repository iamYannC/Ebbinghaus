# Ebbinghaus Illusion Benchmark

## Overview

This project creates a systematic benchmark for evaluating how **both AI vision-language models (VLMs) and human participants** perceive relative size under contextual illusion conditions (the Ebbinghaus illusion). The benchmark generates stimulus images with known ground truth, and supports evaluation by AI models via API or by humans in experimental settings.

The project is designed to be **open-sourced**: the trial table and images can be shared with other researchers for independent analysis and experimentation.

### Core Design Principle: Parameters First, Rendering Second

The **trial table (`trials.csv`) is the single source of truth** for the entire project. Every stimulus image is fully determined by a row in this table. The rendering functions (`draw_trial()`, `render_stimuli()`) are pure consumers â€” they read parameters from the table and produce images. They never generate, modify, or infer parameters.

This means:
- **Researchers control the experiment entirely through the trial table.** To run a custom experiment (e.g., only diagonal layouts, only circles, only Tier 2 trials), a researcher filters or constructs a trial table and passes it to the renderer.
- **The trial table can be created by any means:** our `generate_design()` function, manual construction, programmatic filtering of an existing table, or any external tool that produces the correct schema.
- **Reproducibility is ensured by the trial table itself.** Given the same `trials.csv`, `render_stimuli()` produces identical images. For random trial generation, a `seed` column stores the RNG seed used, so the generation can be reproduced independently.
- **The rendering layer has no opinions.** It does not validate experimental design, enforce balance, or require specific parameter combinations. It renders whatever it receives.

### Modularity Principles

- **`draw_shape.R`** knows nothing about trials or the Ebbinghaus illusion. It draws a single shape on an existing plot/canvas.
- **`draw_trial.R`** takes a single trial row (a one-row data frame or named list with the trials schema) and composes the full stimulus image. It knows about stimulus layout but not about design matrices or evaluation.
- **`verify_trial.R`** computes ground truth from parameters only. It never looks at images. It is deterministic and pure.
- **`classify_tier.R`** assigns a difficulty tier from parameters only. Deterministic and pure.
- **`generate_trial.R`** creates a single trial's parameters, optionally constrained to a target tier. Uses randomization.
- **`generate_design.R`** orchestrates the full experimental design (factorial, fractional, or stratified by tier).
- **`render_stimuli.R`** takes the trial table and produces image files. Rendering is separated from parameter generation.
- **`evaluate.R`** handles all API interaction. Model-agnostic.
- **`analyze.R`** handles all downstream analysis. Operates on the joined trials + evals tables.

## Researcher Workflow

The project supports multiple usage patterns. In all cases, the trial table is the interface between design and rendering.

### Pattern 1: Use the full default design

```r
source("R/generate_design.R")
source("R/render_stimuli.R")

trials <- generate_design(seed = 7042)
render_stimuli(trials)
```

### Pattern 2: Filter an existing trial table

```r
trials <- read.csv("data/trials.csv")

# Only diagonal, circle-only, Tier 2 trials
my_subset <- trials |>
  dplyr::filter(
    orientation == "diagonal",
    test_a_shape == "circle",
    test_b_shape == "circle",
    tier == 2
  )

render_stimuli(my_subset)
```

### Pattern 3: Generate custom trials with constraints

```r
source("R/generate_trial.R")
source("R/render_stimuli.R")

# Generate 100 Tier 1 trials, all horizontal, circles only
my_trials <- purrr::map_dfr(1:100, \(i) {
  generate_trial(
    seed = 5000 + i,
    tier = 1,
    orientation = "horizontal",
    shape_pool = "circle"
  )
})

my_trials <- verify_trial(my_trials)
my_trials <- classify_tier(my_trials)
render_stimuli(my_trials)
```

### Pattern 4: Manual / external construction

A researcher can construct a trial table in any tool (Excel, Python, etc.) as long as it conforms to the trials table schema. Then:

```r
my_trials <- read.csv("my_custom_trials.csv")
render_stimuli(my_trials)
```

### For human experiments

The same trial table and images work for human testing. A researcher presents the rendered images to participants (e.g., via PsychoPy, jsPsych, or a web app), records responses in the same `evals.csv` schema (substituting participant ID for model fields), and uses the same analysis pipeline.

## Phase 1: Stimulus Creation âœ“

**Status: Complete.** All code implemented, tested, and visually validated.

Phase 1 generates the stimulus images that form the benchmark's image bank. It takes a set of trial parameters (shapes, sizes, colors, layout) and renders Ebbinghaus-style illusion images with known ground truth. The output is a `trials.csv` metadata table paired with rendered images in `images/`.

Key properties:
- **Parameters first:** The trial table fully determines every image. Researchers control the experiment by constructing, filtering, or generating trial tables. The renderer is a pure consumer.
- **Ground truth is computed, never manual:** `verify_trial()` derives `true_larger` and `true_diff_pct` from size parameters. `classify_tier()` assigns difficulty tiers. Both are deterministic pure functions.
- **Geometric validity is enforced:** The generator ensures surrounds don't occlude test shapes, adjacent surrounds don't overlap, groups don't intrude on each other, and all shapes contrast against the background. See `config/defaults.R` for tunable parameters.
- **Reproducible:** Every randomly generated trial stores its RNG seed. Given the same seed, `generate_trial()` produces identical parameters.
- **Image naming:** Files are named `<trial_id>_<true_larger>_t<tier>.<ext>` (e.g., `7_equal_t1.png`). Use `strip_answer_from_images()` to create clean copies (`7.png`) before sending to AI models.

### 1.1 `draw_shape()`

Draws a single shape on the current graphics device / ggplot layer.

**Parameters:**

| Parameter | Type     | Values / Range         | Description                                |
|-----------|----------|------------------------|--------------------------------------------|
| `shape`   | character| `"circle"`, `"square"` | Shape to draw                              |
| `size`    | numeric  | `> 0`                  | Radius (circle) or half side-length (square), in plot units |
| `color`   | character| Any valid color        | Border/stroke color                        |
| `fill`    | character| Any valid color or `NA`| Fill color (`NA` = no fill). *Future: patterns* |
| `x`       | numeric  | any                    | Center x-coordinate                        |
| `y`       | numeric  | any                    | Center y-coordinate                        |
| `alpha`   | numeric  | `[0, 1]`              | Opacity. Default `1`.                      |

**Returns:** A ggplot layer (geom) that can be added to a plot.

**Notes:**
- The `size` parameter is defined so that circles and squares of the same `size` value occupy comparable visual area. For a circle, `size` = radius. For a square, `size` = half the side length. This means a circle of `size = r` has area `Ï€*rÂ²` and a square of `size = r` has area `(2r)Â² = 4rÂ²`. They are not identical in area, but they are comparable in "visual extent" (diameter vs side length are both `2r`). This is a deliberate design choice documented here; researchers can account for it in analysis.
- Future extensions: `line_width`, `rotation` (for squares), `pattern_fill`, additional shapes (triangle, hexagon).

### 1.3 `verify_trial()` â€” Ground Truth Verifier

A **pure, deterministic function** that takes trial parameters and returns ground truth.

**Input:** A single row (or data frame) of trial parameters (at minimum: `test_a_size`, `test_b_size`, `test_a_shape`, `test_b_shape`).

**Output:** Adds/overwrites the following columns:
- `true_larger`: `"equal"` if sizes are identical, `"a"` if A is larger, `"b"` if B is larger.
- `true_diff_pct`: `abs(test_a_size - test_b_size) / max(test_a_size, test_b_size) * 100`

**Rules:**
- Comparison is strictly on the `size` parameter. Since `size` is defined consistently across shapes (radius for circle, half-side for square), this is an apples-to-apples comparison of visual extent.
- Uses a floating-point tolerance (e.g., `1e-9`) for the equality check.
- This function has **no side effects**. It does not read images. It does not depend on rendering.

---

### 1.4 `classify_tier()` â€” Difficulty Tier Classifier

A **pure, deterministic function** that assigns a difficulty tier based on trial parameters.

**Input:** A single row (or data frame) of trial parameters.

**Output:** Adds/overwrites `tier` column.

**Tier definitions:**

| Tier | Name              | Condition                                                                 |
|------|-------------------|---------------------------------------------------------------------------|
| `0`  | **Sanity check**  | No surrounds (`surround_a_n == 0` AND `surround_b_n == 0`). Just two shapes. Can the model compare sizes at all? |
| `1`  | **Classic illusion** | Test sizes are equal (`true_larger == "equal"`), surrounds differ in size. Pure illusion condition. |
| `2`  | **Incongruent (conflict)** | Test sizes differ, but surrounds push perception the *wrong* way. The truly larger test stimulus is surrounded by larger context shapes (making it appear smaller). |
| `3`  | **Congruent**      | Test sizes differ, and surrounds reinforce the true difference. The truly larger stimulus has small surrounds (making it appear even larger). |

**Incongruent/Congruent logic:**
- Larger surrounds make the enclosed test shape *appear smaller*.
- So if `true_larger == "a"` and `surround_a_size > surround_b_size`, the illusion *opposes* the truth â†’ **Tier 2 (incongruent)**.
- If `true_larger == "a"` and `surround_a_size < surround_b_size`, the illusion *reinforces* the truth â†’ **Tier 3 (congruent)**.
- If surround sizes are equal (and surrounds exist, and test sizes differ), this is an **unlabeled/neutral** case. We can assign a tier `4` or `NA` for these. They are still useful trials but don't cleanly fit the illusion framework.

### 1.5 `generate_trial()` â€” Constrained Random Trial Generator

Generates a single trial's parameters with optional constraints.

**Key parameters:**
- `seed`: RNG seed for this trial. Stored in the output row's `seed` column. If `NULL`, a random seed is generated and recorded.
- `tier`: If specified, the generator constrains randomization to ensure the output matches this tier (verified by `classify_tier()`).
- `orientation`: If specified, fixes orientation; otherwise random.
- `shape_pool`: Which shapes to sample from. Default: `c("circle", "square")`.
- `size_range`: Range for test stimulus sizes (as proportions of canvas min-dimension). Default: `c(0.03, 0.08)`.
- `surround_size_range`: Range for surround sizes (proportions). Default: `c(0.02, 0.15)`.
- `surround_n_range`: Range for number of surrounds. Default: `c(4, 8)`.
- `canvas_sizes`: Pool of canvas dimensions. Default: `c(512, 768, 1024)`.
- `color_pool`: Pool of border colors (hex). Default: `c("#000000", "#4D4D4D", "#999999", "#4682B4", "#B22222")`.

All size and distance parameters are **proportions of the canvas min-dimension** and are multiplied by `min(canvas_width, canvas_height)` at generation time. This ensures shapes scale appropriately across different canvas sizes. See `config/defaults.R` for all configurable values.

**Logic for tier-constrained generation:**
1. Generate side A parameters randomly within ranges.
2. Based on target tier, generate side B parameters that satisfy the tier's conditions:
   - Tier 0: Zero surrounds for both sides.
   - Tier 1: Set `test_b_size = test_a_size`. Randomize surround sizes such that they differ.
   - Tier 2: Set `test_b_size â‰  test_a_size`. Set surround sizes so the illusion opposes truth.
   - Tier 3: Set `test_b_size â‰  test_a_size`. Set surround sizes so the illusion reinforces truth.
3. Call `verify_trial()` and `classify_tier()` to confirm.

### 1.6 `generate_design()` â€” Design Matrix Builder

Builds a complete trial table for an experiment. **This is one provided strategy for generating trials, not the only way.** Researchers can also:
- Construct a trial table manually (e.g., in a spreadsheet).
- Filter an existing `trials.csv` to a subset.
- Write their own generator using `generate_trial()` in a loop with custom constraints.
- Use any external tool, as long as the output conforms to the trials table schema.

All downstream functions (`verify_trial()`, `classify_tier()`, `render_stimuli()`) accept any data frame with the correct columns, regardless of how it was created.

**Default strategy (our experiment):**
- **Core factorial block**: Cross key variables (true_larger Ã— true_diff_pct levels Ã— surround_size_ratio Ã— orientation). This is the analysis-critical subset.
- **Tier-stratified block**: Ensure minimum representation per tier (e.g., at least N trials per tier).
- **Robustness block**: Vary nuisance parameters (color, canvas size, file format, shape combinations) using Latin-square or random sampling to avoid full factorial explosion.
- **Counterbalancing**: For every trial where A has large surrounds, there should be a matched trial where B has large surrounds (controls for spatial bias).

**Parameters:**
- `seed`: RNG seed for the entire design generation. Stored in each trial row's `seed` column for traceability.
- `n_per_tier`: Minimum number of trials per tier.
- Other configuration parameters for ranges, pools, etc.

**Output:** A complete `trials` data frame, verified and tier-classified.

### 1.7 `render_stimuli()` â€” Batch Renderer

Takes a trial data frame (or any subset of one) and renders each row to an image file.

**Input:** A data frame with the trials table schema (or a subset of its rows). The function iterates over rows and calls `draw_trial()` for each.

**Contract:** `render_stimuli()` has **no opinions about experimental design**. It does not validate balance, tier distribution, or parameter combinations. It renders whatever rows it receives. This is critical for researcher flexibility: a user can filter `trials.csv` to only diagonal-circle-Tier-2 trials and pass that subset directly to `render_stimuli()`.

**Logic per row:**
1. Call `draw_trial()` with the row's parameters.
2. Save to `file_path` in the specified `file_format` at the specified canvas dimensions.
3. Confirm file exists; log any errors.

**Notes:**
- The rendering is deterministic given the same parameters.
- Images are named `<trial_id>_<true_larger>_t<tier>.<ext>` (e.g., `images/1_b_t3.png`, `images/9_equal_t1.png`). This encodes ground truth and tier for researcher convenience. Use `strip_answer_from_images()` (in `R/strip_answer.R`) to create clean copies without answers before AI evaluation.

## Phase 2: Evaluation Pipeline

### 2.1 Prompts Table (`prompts.csv`)

| Column                 | Type      | Description                                             |
|------------------------|-----------|---------------------------------------------------------|
| `prompt_id`            | integer   | PK                                                     |
| `system_prompt`        | character | System-level instruction to the VLM                    |
| `user_prompt_template` | character | User prompt with placeholders (e.g., `{shape}`)        |
| `response_format`      | character | `"free_text"`, `"forced_choice"`, `"json"`             |
| `allows_unknown`       | logical   | Whether "unknown"/"I don't know" is a valid response   |
| `description`          | character | Human-readable label for this prompt variant            |

**Example prompts:**

1. **Minimal forced-choice:**
   - System: `"You are a visual analysis assistant."`
   - User: `"Look at the two central {test_a_shape}s in this image. Which one is larger? Answer with exactly one word: left, right, equal, or unknown."`

2. **Expert persona:**
   - System: `"You are an expert in visual perception and geometric measurement. You are not susceptible to optical illusions. Base your answer only on the true sizes of the shapes."`
   - User: `"Examine the two central shapes. Which central shape has a larger size? Respond with: left, right, equal, or unknown."`

3. **Chain-of-thought:**
   - System: `"You are a careful visual analyst. Think step by step before answering."`
   - User: `"Look at the two central shapes. First, describe what you see. Then, ignoring the surrounding shapes, determine which central shape is truly larger. End your response with ANSWER: left/right/equal/unknown."`

**Note on orientation:** The user prompt should use direction words matching the trial's orientation (`left`/`right` for horizontal, `top`/`bottom` for vertical, `upper-left`/`lower-right` for diagonal). The evaluation pipeline resolves these to `a`/`b` for comparison.

## Phase 3: Analysis

### 3.1 Joined Table

```
trials LEFT JOIN evals ON trial_id
       LEFT JOIN prompts ON prompt_id
```

Derived columns:
- `correct`: `response_larger == true_larger` (logical)
- `illusion_susceptible`: On Tier 1 trials, `response_larger != "equal"` (logical)
- `illusion_direction`: On Tier 1 trials, which side the model chose (indicates directional bias from the illusion)

#### Feature engineering examples

These columns are not pre-computed in the trial table to keep stored data lean. Researchers can derive them as needed:

```r
trials_enriched <- trials |>
  mutate(
    # Do the surrounds match the test shape?
    surround_a_matches_test = (surround_a_shape == test_a_shape),
    surround_b_matches_test = (surround_b_shape == test_b_shape),
    # Surround size ratio (how much bigger/smaller are surrounds vs test?)
    surround_a_ratio = surround_a_size / test_a_size,
    surround_b_ratio = surround_b_size / test_b_size,
    # Color match between test and surround
    color_a_matches = (test_a_color == surround_a_color),
    color_b_matches = (test_b_color == surround_b_color),
    # Same shape on both test sides?
    test_shapes_match = (test_a_shape == test_b_shape)
  )
```

### 3.2 Key Analyses

| Analysis                    | Description                                                                                         |
|-----------------------------|-----------------------------------------------------------------------------------------------------|
| **Overall accuracy**        | Proportion correct by model, by tier                                                                |
| **Psychometric curves**     | Accuracy as a function of `true_diff_pct`, per model. The signature plot of this benchmark.         |
| **Illusion susceptibility** | On Tier 1 (equal) trials: proportion answering non-equal. Higher = more susceptible.                |
| **Illusion direction**      | On Tier 1 trials: does the model consistently pick the side with small surrounds as "larger"? (Classic Ebbinghaus response) |
| **Spatial bias**            | Across all trials: does the model favor A or B regardless of truth?                                 |
| **Congruency effect**       | Accuracy on Tier 2 (incongruent) vs Tier 3 (congruent). Difference = illusion strength.             |
| **Prompt effects**          | Accuracy by `prompt_id`. Does chain-of-thought help? Does the "expert" persona reduce illusion?     |
| **Temperature effects**     | Accuracy by temperature. Is deterministic (`temp=0`) better?                                        |
| **Format effects**          | Accuracy by `file_format`. Do SVGs help?                                                            |
| **Confidence calibration**  | Among trials where confidence is reported: is higher confidence associated with higher accuracy?     |
| **d-prime (sensitivity)**   | Signal detection metric separating perceptual ability from response bias, computed per model.        |

### 3.3 Key Visualizations

1. **Psychometric curve**: x = `true_diff_pct`, y = P(correct), color = model. Include error bars.
2. **Confusion matrix heatmap**: true_larger vs response_larger, per model.
3. **Illusion susceptibility bar chart**: proportion "fooled" on Tier 1, by model.
4. **Congruency effect plot**: accuracy on Tier 2 vs Tier 3, by model (paired).
5. **Spatial bias plot**: proportion choosing A vs B, by model and orientation.
6. **Prompt comparison**: accuracy by prompt variant, faceted by tier.
